Metric definitions: one query surface for every metric

The rule: every metric exposes its aggregate under the same column name, value, no matter what it wraps underneath. Right now it doesn't, and that's the whole reason this doc exists.

What a metric is for us

A metric is a definition layered on top of our semantic models. It picks one measure to aggregate, one time field to filter and bucket on, a grain, and a set of dimensions to slice by. The semantic model does the heavy lifting (the joins, the physical columns); the metric just names what we care about and how to cut it.

Two live examples:

kind: metric
name: on_time_delivery_rate
measure: shipments.on_time_pct
ts: shipments.delivered_at
granularity: month
description: Share of shipments delivered within the promised window

dimensions:
  - name: lane
    ref: shipments.origin_dest_lane
  - name: carrier
    ref: shipments.carrier_name
kind: metric
name: fleet_utilization
measure: vehicles.active_ratio
ts: vehicles.reading_at
granularity: day
description: Fraction of the fleet in active service

dimensions:
  - name: depot
    ref: vehicles.home_depot
  - name: vehicle_class
    ref: vehicles.class

Notice these two disagree about what ts even means. on_time_delivery_rate buckets by a delivery window (a span, anchored to month end). fleet_utilization is a point-in-time reading taken each day. Same field name, different time semantics. Hold that thought.

How we query it

select MEASURE(measure), lane
from on_time_delivery_rate
where <ts filter>
  and carrier = 'temp'
group by 2;

MEASURE() is the virtual aggregation function. It resolves the metric's declared aggregate and rolls it up over whatever dimensions survive the group by. This is the surface every one of us touches, and it's the same shape for every metric, which is the point of having a metric layer at all.

The problem

Look at that query again: MEASURE(measure). We're aggregating a column called measure. The name restates that it's a measure and tells you nothing about the number coming back. Is it a rate? A count? Dollars? The column name shrugs.

It's an artifact of the definition. measure: is the fixed keyword in the metric kind, so the aggregate gets exposed under the literal word "measure." Every metric, every query, reads MEASURE(measure). It parses fine and it's completely uninformative, the SQL equivalent of a variable named variable.

ts doesn't have this problem, for what it's worth. It's a neutral name we already alias well, and the two examples above show why keeping it neutral is correct: it has to cover both a delivery window and a daily instant. Grain and anchor live in the definition (granularity, and a period_anchor field if we want to make start-vs-end explicit), not in the column name. ts stays.

The fix

Rename the keyword from measure: to value:.

measure: shipments.on_time_pct
value: shipments.on_time_pct

The query becomes:

select MEASURE(value), lane
from on_time_delivery_rate
where <ts filter>
  and carrier = 'temp'
group by 2;

MEASURE(value) reads as what it is: aggregate the value of this metric. One word changed. ts untouched. The convention we get for free is that every metric now exposes exactly two fixed names, value and ts, regardless of which semantic model or column sits underneath. Nothing to memorize per metric.

Why value and not something else

Options considered:

metric_value — accurate but redundant. You're always selecting from <metric_name>, so the metric is already in scope. The prefix adds characters and says nothing new.

result — collides with a computation-status reading. "Result" sounds like it could tell you whether the calc succeeded, not what the number is.

amount / qty — wrong for rates and ratios. on_time_delivery_rate isn't an amount of anything.

value — neutral across every metric type (rate, count, ratio, dollar figure), short, and reads correctly inside MEASURE(). The metric name carries the "of what," so value carrying the "the number" is all we need.

Why ts and not something else

We already use ts, so this is a defense of keeping it, not a change. It came up because a few of the alternatives read cleaner in isolation, but each one lies about at least one of our metrics.

period — implies a duration, a span with a start and an end. Fine for on_time_delivery_rate (a monthly window), wrong for fleet_utilization (a daily instant, not a range). Naming a point-in-time reading period misleads.

period_start — worse. It hard-codes a start anchor into the name, but on_time_delivery_rate anchors to month end while fleet_utilization anchors to a daily reading. Bake _start in and the name is wrong for half the metrics. Anchor is real metadata; it belongs in the definition (a period_anchor field), not smuggled into a column name.

time — fully neutral on duration-vs-instant, which is what we want, but it's a reserved or typed keyword in several engines (Postgres, MySQL, others). You'd be quoting it or hitting parser friction in generated SQL. Not worth the constant escaping.

ts — neutral, short, collision-free everywhere, and it makes no claim about whether the number describes a span or a moment. That neutrality is the feature: one metric's ts is a delivery window, another's is a daily snapshot, and the name is honest about both because it commits to neither. Grain and anchor carry the real semantics in the definition.

The only knock on ts is that it whispers "raw timestamp" when some metrics are bucketed. That's cosmetic, and granularity in the same YAML block already tells the reader it's bucketed. Keeping ts costs us nothing and dodges the reserved-word problem time would hand us.

The change

It's a rename of the keyword plus a mechanical pass over existing definitions. Cheap to handle, no query rewrites beyond swapping the word.

The contract Every metric exposes value and ts. Always those two names. What they point to is the metric's business; what you type to query them never changes.

  - name: carrier
    ref: shipments.carrier_name

ts: vehicles.reading_at

dimensions:
  - name: depot
    ref: vehicles.home_depot
  - name: vehicle_class
    ref: vehicles.class

value: and ts: are the only two fixed keywords that point at columns. Everything else (granularity, dimensions, and their name/ref pairs) is unchanged.

Placeholders to swap before circulating: the example column refs (shipments.on_time_pct, vehicles.active_ratio) are in our style but invented, and the period_anchor mentions assume that field exists. If it doesn't, soften both to "the definition."
