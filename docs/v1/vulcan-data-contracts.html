data contract data product — field guide for vulcan

Two words the industry keeps confusing — and a spec for each one.

A data contract is the interface agreement for a single dataset. A data product is the discoverable, owned package built from one or more contracts. This guide covers the standards that formalize both, and maps every field onto what Vulcan already has — and doesn't.

Part A · 01

§1 Where "data product" comes from

Before either spec existed, Zhamak Dehghani defined the term in her 2019 data mesh articles on martinfowler.com. Her second principle of data mesh — "data as a product" — treats each domain's dataset the way a software team treats a shipped product: something built for named consumers, held to a quality bar, and supported after release rather than dumped into a lake and forgotten.

Her original text names six characteristics a domain dataset needs before it earns the label "product." These six — not the "PACT" or "DATSIS" acronyms sometimes attributed to her — are the actual source text, and everything ODCS and ODPS formalize below traces back to one of them.

1
Discoverable

Listed in a registry or catalogue with its own metadata, so consumers can find it without asking around.

2
Addressable

Reachable at a stable, global address a program can hit — not a moving target buried in a folder structure.

3
Trustworthy & truthful

Backed by an explicit SLO on truthfulness, so no one has to double-check it before use.

4
Self-describing

Understandable and consumable without hand-holding from the team that built it.

5
Interoperable

Follows shared standards and harmonization rules so it can be correlated with data from other domains.

6
Secure

Access control applied at the granularity of the product itself, not just the underlying warehouse.

Dehghani's follow-up article goes further: a data product assumes the role of an architectural quantum — a term she borrows directly from Ford, Parsons & Kua's Building Evolutionary Architectures ("the smallest unit of architecture that can be independently deployed with high functional cohesion") — bundling code, data-and-metadata, and infrastructure into one independently deployable unit. The idea: a data product isn't just a table, it's closer to a small service. ODPS doesn't go that far, but its inputPorts/outputPorts model is a direct, if lighter-weight, descendant of the same idea.

Part A · 02

§2 Contract vs. product: the granularity split

Every spec below sorts into one of two buckets, and mixing them up is the single most common confusion in this space (helped along by the fact that two entirely unrelated standards both go by the initialism "ODPS" — more on that in §5).

Data contractData product
UnitOne dataset, table, topic, or fileOne or more contracts, bundled
Answers"What does this dataset look like, and what do I promise about it?""What can I discover, subscribe to, and get support for?"
CarriesSchema, types, quality rules, SLA, pricing, serversPorts (each pointing at a contract), team, support channels
AnalogyAn OpenAPI document for one endpointA developer-portal listing for a whole API

In the Bitol ecosystem this split is explicit and literal: an ODPS product document doesn't embed a schema at all — it just points, via contractId, at one or more ODCS contract documents that do.

Part A · 03

§3 Open Data Contract Standard (ODCS)

ODCS is the schema-level agreement between a producer and its consumers — the closest thing this space has to OpenAPI. It's governed by Bitol, a Linux Foundation AI & Data project, currently at Incubation stage.

History

PayPal's internal data-contract template, built by Jean-Georges Perrin's team for PayPal's own data-mesh rollout, was open-sourced in May 2023. Four months later it was contributed into the newly-formed Bitol project and renamed the Open Data Contract Standard. Bitol was jointly stood up by the nonprofit AIDA User Group and Linux Foundation AI & Data on November 30, 2023, and now hosts both ODCS and ODPS under one TSC, chaired by Perrin.

VersionDateNote
v2.2.02023-07-27PayPal template renamed ODCS

v3.0.22025-03-31Patch release
v3.1.02025-12-08Current. Adds relationships (FKs), reworks team, deprecates dataProduct

Two version numbers, not one

apiVersion is the spec version (v3.1.0); version is the contract's own semver, bumped independently as the underlying dataset evolves. status carries the lifecycle: proposed → draft → active → deprecated → retired.

Fundamentals

status: active

tenant: ClimateQuantumInc

description:
  purpose: Views built on top of the seller tables.
  limitations: Cannot be used in conjunction with days with full moons.
  usage: Twice a day, preferable before meals.

tags: ['finance']

Schema — objects, properties, quality

ODCS calls a table/topic/document an object and a column/field a property. Quality rules attach at either level.

schema:
  - id: tbl_obj
    name: tbl
    logicalType: object
    physicalType: table
    physicalName: tbl_1
    description: Provides core payment metrics
    properties:
      - id: rcvr_id_prop
        name: rcvr_id
        primaryKey: true
        primaryKeyPosition: 1
        businessName: receiver id
        logicalType: string
        physicalType: varchar(18)
        required: false
        unique: true
        classification: restricted
        quality:
          - id: rcvr_id_no_nulls
            type: library
            metric: nullValues
            mustBe: 0
            unit: rows

logicalType is one of string, date, timestamp, time, number, integer, object, array, boolean. Quality type is one of library (built-in metrics like nullValues, duplicateValues, rowCount), text, sql, or custom (a named engine like Soda, with an inline implementation block). Every quality rule also carries a dimension: accuracy, completeness, conformity, consistency, coverage, timeliness, uniqueness.

New in v3.1.0: a relationships block lets one object declare a foreign key against another, using dot-path notation for composite keys — the first time ODCS has modeled cross-table structure rather than treating every object as an island.

Team, support, SLA, price

team:
  id: tsc_team
  name: TSC
  members:
    - username: daustin
      role: Owner
      name: David Austin
      dateIn: 2022-10-01

support:
  - channel: data-payments
    tool: slack
    scope: interactive

slaProperties:
  - property: latency
    value: 4
    unit: d
    element: tab1.txn_ref_dt
    scheduler: cron
    schedule: "0 30 * * *"

price:
  priceAmount: 9.95
  priceCurrency: USD
  priceUnit: megabyte

Valid SLA property values read like a QoS periodic table: availability, throughput, errorRate, generalAvailability, endOfSupport, endOfLife, retention, frequency, latency, timeToDetect, timeToNotify, timeToRepair. As of v3.1.0, team is an object with a members array, not the flat array older contracts used — a breaking-but-mechanical migration if Vulcan ever needs to read older contracts.

Correction worth flagging

The dataProduct field that used to sit at the top of an ODCS document — a bare string naming the parent product — is deprecated as of v3.1.0. Don't build a new integration around it. The relationship between a contract and its product now flows the other way: the product (ODPS) points at the contract, not the reverse.

Part A · 04

§4 Data Contract Specification (datacontract.com)

A second, independently-invented spec covering the same ground. Created by Jochen Christ and Dr. Simon Harrer of the German consultancy INNOQ, publicly launched September 2023 — unrelated to the PayPal/Bitol lineage. Its YAML shape is recognizably similar to ODCS (an info block, keyed servers, models with typed fields, inline SQL-based quality checks, ISO-8601 servicelevels), and it shipped with a genuinely useful CLI — datacontract-cli — with commands like test (run quality checks against a live connection), export/import (to dbt, Avro, JSON Schema, Terraform, and more), and breaking (diff two contract versions and flag consumer-breaking changes).

This matters for anything Vulcan builds today

On December 8, 2025 — the same day ODCS v3.1.0 shipped — the Data Contract Specification's own maintainers (by then operating as Entropy Data) added a formal deprecation notice to their README:

"With the release of the Open Data Contract Standard v3.1.0, we deprecate the Data Contract Specification in line with our commitment to focus on a single industry standard for data contracts. […] The Data Contract Specification will be supported in Data Contract CLI and Entropy Data until the end of 2026."

This isn't a rebrand or a merge — it's a sunset. The two rival camps converged on ODCS as the one standard. Build against ODCS, not the Data Contract Specification. If Vulcan ever needs to import a legacy datacontract.yaml, treat it purely as a migration path (the CLI even ships datacontract export --format odcs for exactly this).

Part A · 05

§5 Open Data Product Standard (ODPS)

Disambiguation — two unrelated specs share this acronym

There is a second, older, and completely unrelated "Open Data Product Specification" from the independent Open Data Product Initiative (Moilanen & Niilahti, opendataproducts.org, a separate Linux Foundation project since 2024). It has its own pricingPlans, dataQuality, dataHolder, and license fields, and is not designed to pair with ODCS. Everything below is Bitol's ODPS — the one built as ODCS's sibling — since that's the one Vulcan's own quality-dimension taxonomy already echoes (see §8).

ODPS v1.0.0 shipped September 24, 2025 under the same Bitol/TSC governance as ODCS. Structurally it's much thinner than ODCS — a product document doesn't carry a schema, quality rules, SLA, or pricing at all. It's purely a wrapper: identity, ports, team, support.

apiVersion: v1.0.0
kind: DataProduct
id: a1b2c3d4-...
name: customer-data-product
status: active
domain: customer

outputPorts:
  - name: rawtransactions
    description: "Raw Transactions"
    type: tables
    version: 2.0.0
    contractId: c2798941-1b7e-4b03-9e0d-955b1a872b33
    tags: ["transactions"]

team:
  members:
    - username: jchen
      role: owner

support:
  - channel: customer-data
    tool: slack
    scope: interactive

The link to a contract is nothing more than contractId + version — an opaque pointer your platform resolves against its own ODCS store. There's no embedded schema, and — this trips a lot of first reads — no SLA or pricing field on the product at all. Those stay exactly where ODCS put them, inside the referenced contract. A product can also declare inputPorts (what it consumes) and managementPorts (discoverability/observability/control endpoints), but outputPorts is the one that matters for "what does this product expose."

§6 How the incumbents handle this today

Worth knowing mainly so Vulcan doesn't reinvent a worse version of something a bigger vendor already tried and abandoned.

VendorWhat they call itNative "data contract"?
DatabricksNo formal object; "one data product = one Unity Catalog schema" by convention. Metric views (2025) are a separate semantic-layer feature.No — blog explicitly recommends bringing your own ODCS YAML
AWS DataZoneFormal DataProduct API object bundling one or more assets; subscription/approval workflow governs access.No native term; Glue Data Quality scores surface in the catalog instead
Snowflake"A data product is the share or the app attached to your listing" — official definition.No — own quickstart delegates contracts to dbt's model-contracts feature
dbtexposures (lightweight downstream-dependency declarations) + model contracts (enforced column name/type/constraint checks on one model)Partial — schema-only, no SLA/quality/pricing

The pattern is consistent: every major warehouse vendor has a "data product" concept (a governed, discoverable bundle), and none of them have shipped a native "data contract" object — they either point at ODCS directly or leave the gap to dbt. That's a reasonable signal that ODCS is genuinely becoming the shared substrate, not just Bitol's aspiration.

§8 What Vulcan already has

The honest starting point: Vulcan's core already implements a surprising amount of contract-shaped metadata, largely because it inherited a SQLMesh-style model layer. The gaps are almost entirely at the product layer, not the schema layer.

ConceptWhere it livesStatus

Grain / primary keysModelMeta.grainsImplemented
Logical schema (dimensions, measures, joins)sqlmesh/core/model/semantic.py, loaded from semantics/*.ymlImplemented

Owners, stewardship, access policy, SLA, compliance, linksDocumented in readings/DP-REAMDE/product_authoring_guide.md onlyDesign doc — no Pydantic model, no loader, no API

The single most interesting finding: Vulcan's QualityRule.dimension enum already borrows its taxonomy from the Bitol quality-dimension vocabulary. That's a strong hint that whoever built the DQ layer was already reading these specs — emitting ODCS quality[] blocks from that data is close to a direct field copy, not a redesign.

§9 Gap analysis

What ODCS/ODPS expect that Vulcan has no field for at all — the honest list, not a sugar-coated one.

Missing conceptSpec fieldNotes
Multi-party ownership / stewardship rolesODCS team, ODPS teamVulcan's owner is one string. Authoring-pack doc proposes owners[] + role-keyed stewardship, unimplemented.
SLA (freshness, availability, latency)ODCS slaProperties[]Vulcan has cron (intent) but nothing that publishes an SLO or measures it against reality.
Access classification / PII handlingODCS property-level classificationVulcan has column masking (column_mask_expressions_) — enforcement exists, but no declarative classification label to publish.
Compliance metadata (regimes, residency, retention)Not in ODCS/ODPS core — a custom-property patternAuthoring-pack doc proposes this; both specs leave room for it via customProperties.
Support channelsODCS support[], ODPS support[]No structured equivalent; only free-text links in the aspirational doc.
Product-level ports (input/output)ODPS inputPorts / outputPortsVulcan's dependency graph captures the equivalent relationships implicitly, but nothing packages a product's exposed surface as ports.
Foreign-key relationships between objectsODCS relationships[] (new in v3.1.0)Vulcan's semantic joins already carry this information logically — needs translation, not new data.

Part B · 11

§11 Proposed API surface

Additive to the existing /metadata family in api/endpoints/metadata.py — nothing here requires changing current responses.

GET /metadata/contracts List every model available as an ODCS contract: id, name, current version, status.
GET /metadata/contracts/{model_name} Emit one model as a full ODCS v3.1.0 document. Content-negotiated: Accept: application/odcs+yaml;version=3.1.0 for YAML, application/json for the JSON-Schema-validated equivalent.
GET /metadata/contracts/{model_name}/versions History of contract versions for one model, keyed to SQLMesh snapshot fingerprints (see below).
GET /metadata/product Emit the whole project as one ODPS v1.0.0 document — one outputPort per published model, each contractId resolving back to /metadata/contracts/{model_name}.

Versioning: reuse what SQLMesh already computes

This is the one place a genuinely novel idea, rather than a field copy, pays off. SQLMesh's planning phase already classifies every model change as breaking or non-breaking via content fingerprinting, in order to decide whether a virtual-environment rebuild is needed. That's precisely the signal datacontract-cli breaking exists to compute after the fact for the Data Contract Specification — Vulcan already has it, for free, at plan time.

Proposal

Bump a contract's ODCS version major/minor/patch component using SQLMesh's existing breaking-change classification at the moment a plan is applied, instead of asking model authors to hand-maintain a semver. status transitions (draft → active → deprecated) follow the model's own lifecycle state, not a separate manual field.

New schema classes required

A thin translation layer, not a new subsystem — schema/metadata.py already has 90% of the source data:

# schema/contract.py — new
class OdcsProperty(BaseModel):
    id: str
    name: str
    logicalType: str
    physicalType: str
    businessName: str | None
    primaryKey: bool = False
    classification: str | None
    quality: list[QualityRule] = []

class OdcsContract(BaseModel):
    apiVersion: str = "v3.1.0"
    kind: str = "DataContract"
    id: str
    name: str
    version: str
    status: str
    schema_: list[OdcsSchemaObject] = Field(alias="schema")
    team: OdcsTeam | None
    slaProperties: list[OdcsSlaProperty] = []

# schema/product_spec.py — new
class OdpsOutputPort(BaseModel):
    name: str
    type: str
    version: str
    contractId: str

class OdpsProduct(BaseModel):
    apiVersion: str = "v1.0.0"
    kind: str = "DataProduct"
    id: str
    name: str
    domain: str | None
    outputPorts: list[OdpsOutputPort]
    team: OdcsTeam | None

§12 Build order

  1. ODCS emitter for one model. A pure function, Model → OdcsContract, covering fundamentals + schema + quality — everything in §10's first table already exists in ModelMeta. No new stored fields required.
  2. GET /metadata/contracts/{model_name}. Wire the emitter into the existing FastAPI app; content-negotiate YAML vs. JSON.
  3. Version derivation. Hook contract version/status to snapshot fingerprint + breaking-change classification, so the field is computed, not authored.
  4. ODPS emitter for the whole project. One outputPort per published model; trivial once step 1–2 exist, since contractId is just the model's own contract id.
  5. Implement the authoring-pack fields for real (owners, stewardship, accessPolicy, sla, compliance, links) as an actual Pydantic model loaded from project config, closing the gap in §9 — this is the only step that needs genuinely new data, not just translation of existing data.
  6. GraphQL parity. Mirror both new REST resources as GraphQL query types alongside the existing SemanticSchema/SemanticEntry types in graphql/src/types/semantic.ts.
