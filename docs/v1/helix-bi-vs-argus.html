AI with Argus: Bringing Meaning to Data

Storing, moving, and documenting data is no longer the hard part. The harder problem is meaning: how terms, assets, and concepts are understood across teams, products, and workflows. Without that layer, discovery stays shallow, reasoning breaks, and AI systems can only operate on the surface of what the data contains.

01

An Example of Improved Reasoning with Argus

Helix Industrial — B2B manufacturer of electronics; parts embedded in third-party medical devices (MRI scanners, surgical robots). 56 strategic accounts hold most of the revenue.

February 2026: a supplier substitutes a sub-component. It passes incoming inspection but drifts under operating conditions. Propagation over three months — incoming-quality drift → production rework → defect-rate increase → pullback by five strategic medical customers → receivables aging. Each signal sits in a separate data product; read in isolation, each stays within control limits.

The CRO's question is a business question, not a metric. The data spans supplier, production, quality, delivery, carrier, receivables, and revenue — no single owner, no dashboard scoped to it.

Why better documentation doesn't close the gap

The AI case above uses an agent over table and column metadata — names, descriptions, basic schema. The obvious objection: that's a weak baseline. Write better documentation and the gap closes.

It doesn't. The difference between metadata and an ontology is not documentation quality — it's structure. Documentation is prose. The ontology is executable.

A perfectly written schema could tell an agent that receivables_identity_coverage_pct exists and what it measures. It cannot tell the agent that this metric is the coverage measure for the identity bridge a prior answer's AR branch depended on — and therefore what to read when the question is "how confident should I be?" That relationship has to be represented as structure. Prose cannot be traversed; a graph can.

This is why Q6 is the sharpest test in this document. Every gap in Q3–Q5 could in principle be narrowed with richer documentation. Q6 cannot — reporting branch-specific confidence requires identity paths and their coverage measures to be encoded as structure. There is no documentation equivalent to that query.

Why the ontology produces this plan — the reasoning gap

The concept Revenue at Risk carries a note that open-order exposure is only the visible surface, plus dependency edges to customer value, pullback/churn, five operational-friction sub-concepts, and financial distress. Each edge is grounded to specific measures across seven data products, and the identity bridges let every branch resolve to the same customers. The agent inherits a map from the phrase "walking revenue" to eight evidence branches — it does not have to know the domain, only to read the graph.

Metadata has none of this. "Walking" lexically matches the metric name forward_revenue_exposure_open_orders, and with no edges to follow, the plan stops there — the ≈$3.37M visible slice, which by the answer key misses roughly 80% of the at-risk base. The reasoning does not fail; it has nothing to traverse.

The gap is two-part: meaning (walking ≠ exposure) and grounding (meaning → measures across DPs). Remove either and the plan collapses back toward the single-metric answer.

Why the ontology produces this plan — the reasoning gap

The concept silently-at-risk customer is defined as Active + strategic tier + priority industry + at least two pullback shapes, and each shape is grounded to a measure in a different data product. Building the cohort therefore means intersecting four DPs and explicitly excluding the already-lost — a rule that only exists in the graph.

Metadata offers one literal metric that reads like the question: recent_nonactive_customer_count. So "losing customers" resolves to accounts already flagged nonactive — the 15 Inactive/Churned in the wrong industries — and the five active, still-savable accounts never enter the plan. The five pullback shapes differ per customer, so no single measure would have found them anyway.

The gap is the composite cohort definition (the ≥2-of-4 rule spanning four DPs) and the exclusion step. Both are relationships, and relationships are what metadata does not carry.

Why the ontology produces this plan — the reasoning gap

The graph encodes a causal hypothesis as edges — supplier drift → product defect → customer pullback — and grounds the chain through customer_traceable_supplier_issue_count, which links a supplier and material back to specific customers. The agent can therefore walk from the visible symptom to the upstream cause across four DPs, and test lateness against quality to name the mechanism.

Metadata has no causal edge and no representation of the supplier → material → work-order → customer path. So the visible symptom lexically matches internal fulfillment metrics, and the answer stops at a handoff redesign — the wrong direction and the wrong intervention. It answers a question one hop from the one asked.

The gap is the causal relationship plus the cross-DP traceability chain. A metric can be looked up; a direction has to be encoded.

Why the ontology produces this plan — the reasoning gap

The graph names the identity paths each earlier answer relied on, and every path has a published coverage measure. Confidence is therefore a query, not a judgment: read the four coverage percentages, restrict to Strategic accounts, and mark any branch resting on a thin bridge as extrapolated.

Metadata has no representation of identity paths at all — it lists tables and columns, not how entities resolve between them. With nothing to measure, the agent cannot say which parts of its answer are grounded and falls back to a generic disclaimer that is equally true of any answer.

The gap is the explicit identity paths and their coverage measures. The same structure that let the earlier questions cross DPs is what lets this one report how far it can be trusted. This is the one gap no improvement in documentation quality can close — the relationship between a prior answer's branches and the bridges they depended on must be represented as structure to be queryable at all.
